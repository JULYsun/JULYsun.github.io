数据采集与清洗工具

· 2019/07/22 · # Ixd # 工具应用

数据研究的上游需求——数据的获取与清洗等预处理工作往往占据了分析过程80%的时间。数据采集与清洗工具采用半自动化云端协作的方式进行定向采集，并为非数据科学家降低了数据清洗门槛和时间消耗。这里介绍项目中的三个设计片段：1、任务流程的引导；2、数据处理结果展示与交互；3、撤销和返回状态的补充。

1 项目介绍

1.1 项目属性

  • Master/Datail Interface

2 项目设计

2.1 前言

在传统桌面端的数据处理软件中，界面可以展示全量数据，每一步数据清洗操作后数据实时发生变化，对用户来说非常直观，也有很高的操作自由度。然而当数据体量较大时，Web 端的环境很难完成全量数据的传输和展现。因此，由后台反馈需要用户操作的样本数据，并充分提供用户决策信息的交互方式。

基于以下的功能相似性，我们借鉴了类似计算机清理加速软件的设计模式：

  • 线性任务。清理软件的线性操作与「阶段」相似。
  • 提供决策信息。在“清理垃圾文件”功能中需要展示不同类型的垃圾文件和详细信息，帮助用户作出是否删除的判断。

2.4 返回与撤销状态的补充

在产品设计中，操作返回和撤销也是必要的组成部分，在Web环境复杂流程产品中，返回和撤销逻辑同样与服务器端的保存状态密切相关。在设计结果在用户层面需要满足心智模型，在产品层面需要统一的跳转逻辑。
一次数据采集清洗任务包括课题建立、任务设置、数据补缺、数据替换、数据去重共五大步骤，在设计时容易陷入设计细节。由于数据补缺、替换、去重三大操作的相似性（设置-扫描-修改-生成），我们将它们定义为「阶段」。在结构梳理中，将「阶段」作为最小单元。

并在阶段之间加入完成页面，用于承载运行结果和用户操作（继续、导出、返回）。而返回按钮直接跳转至上一阶段的完成页面（回滚一个版本的数据表）：1、方便用户连续返回至前一个数据状态；2、出现数据的运行结果，让用户了解当前状态，对接下来的操作有准确的心理预期。

4 项目总结

数据采集与清洗工具是一次较为复杂的产品设计尝试。我们将一般的设计流程进行解构并应用在多个层次的设计工作中：

当前项目已应用于某高校科研数据研究项目支持中，相比于此前的解决方案，做到了：

  • 整合定向采集需求，防止重复开发工作，降低了高校项目中开发人员的工作量。
  • GUI化的操作界面，满足基本数据清洗需求，降低使用者的专业知识门槛。
  • 成为高校项目的常规产品，丰富产品矩阵。并使产品持续优化扩展成为可能。
